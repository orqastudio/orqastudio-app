## Problem Statement

OrqaStudio is mid-migration: the artifact framework (milestones, epics, tasks, ideas, plans, lessons, research) was designed recently, but significant historical work predates it. That work lives in `docs/` without artifact linkage. Meanwhile, the app can only browse 7 of 12 artifact types — the core agile types (milestones, epics, tasks, ideas) have no UI at all.

If we backfill content without building viewers, the artifacts exist but are invisible. If we build viewers without backfilling, the viewers are empty. Both must move forward together.

## Migration Strategy

### Principle: Markdown First, Viewers Follow

The artifact framework is markdown-first. `.orqa/` files with YAML frontmatter are the source of truth — they work in the CLI, in governance scanning, and in any text editor. UI viewers are display layers on top.

This means we can **backfill content now** and build viewers incrementally. The backfilled artifacts are immediately useful for governance, planning, and CLI work even before the UI catches up.

### Sequencing: Three Waves

#### Wave 1: Backfill Historical Content (no code changes needed)

Populate existing artifact types with historical data. These artifacts are browsable in the CLI and become visible the moment viewers are built.

| Action | Details | Risk |
|--------|---------|------|
| **1a. Research frontmatter** | Add YAML frontmatter to all 9 research docs linking them to the architecture decisions they produced | None — additive |
| **1b. Link orphaned plans** | Connect `agent-governance.md` and `provider-abstraction.md` to their epics, or archive them | None — metadata only |
| **1c. Retroactive epics for completed work** | Create `EPIC-NNN` (or similar) for Phase 1 Scaffold and Phase 2b Governance Bootstrap with `status: done` | Low — historical record |
| **1d. Architecture decisions as research links** | Add `produces_decisions` frontmatter to research docs mapping to AD numbers | None — additive |

#### Wave 2: Build Agile Artifact Viewers (code changes)

Add the missing backend + frontend for milestones, epics, tasks, ideas. This is the [EPIC-005](EPIC-005) (Artifact Browser) work from the roadmap.

| Action | Details | Depends On |
|--------|---------|------------|
| **2a. Backend readers** | Add `scan_*_tree` and `read_*` functions for milestones, epics, tasks, ideas in `artifact_reader.rs` | Wave 1 (content to display) |
| **2b. Tauri commands** | Add `milestone_tree_scan`, `epic_tree_scan`, `task_list`, `idea_tree_scan` + read commands | 2a |
| **2c. Store bindings** | Add load methods to artifact store | 2b |
| **2d. Viewers** | Create MilestoneViewer, EpicViewer, TaskViewer, IdeaViewer with structured frontmatter rendering | 2c |
| **2e. Sidebar navigation** | Add activity bar items for milestones, epics, tasks, ideas | 2d |
| **2f. Lesson viewer** | Build proper LessonViewer, wire to existing lesson activity | Standalone |

#### Wave 3: Cross-Linking & Navigation (code changes)

Make artifacts navigable — clicking an epic shows its tasks, clicking a task shows its epic, clicking research shows what decisions it produced.

| Action | Details | Depends On |
|--------|---------|------------|
| **3a. Internal artifact links** | MarkdownRenderer resolves `[EPIC-001](EPIC-001)`, `[TASK-003](TASK-003)`, `[AD-007](AD-007)` as clickable links to their artifacts | Wave 2 |
| **3b. Relationship sidebar** | When viewing an epic, show related tasks, plans, research, decisions in a sidebar panel | Wave 2 |
| **3c. Roadmap view** | Interactive roadmap showing milestones → epics → tasks hierarchy (replaces monolithic doc) | Wave 2 |

### 1d. Research → Decision Links

The architecture decisions doc (`docs/architecture/decisions.md`) contains [AD-001](AD-001) through [AD-017](AD-017). Each decision was informed by research. The frontmatter links in 1a create the forward reference (research → decision). For the reverse (decision → research), we have two options:

**Option A: Keep decisions as a monolithic doc** — Add a "Research Source" column to the decisions table. Lightweight, no new artifact type.

**Option B: Individual decision artifacts** — Create `.orqa/decisions/AD-NNN.md` files with frontmatter linking back to research. Richer, but more work and requires a new artifact type + viewer.

**Recommendation:** Option A for now (Wave 1), Option B as a future enhancement (Wave 3 or beyond). The monolithic doc works fine for reference; individual artifacts add value when we want to navigate decisions in the app.

## How Migration and Development Coexist

### The Interleaving Plan

```
Wave 1 (backfill)          Wave 2 (viewers)              Wave 3 (linking)
─────────────────          ─────────────────              ─────────────────
1a. Research frontmatter ──┐
1b. Orphaned plans ────────┤
1c. Retroactive epics ─────┤
1d. Decision links ────────┘
                           ├─ 2a. Backend readers
                           ├─ 2b. Tauri commands
                           ├─ 2c. Store bindings
                           ├─ 2d. Viewers
                           ├─ 2e. Sidebar nav
                           └─ 2f. Lesson viewer
                                                         ├─ 3a. Internal links
                                                         ├─ 3b. Relationship sidebar
                                                         └─ 3c. Roadmap view
```

**Key principle:** Wave 1 runs first and produces no code changes — only markdown files. It can be done in a single session. Wave 2 is [EPIC-005](EPIC-005) (Artifact Browser) from the roadmap, which is already a P2 dogfooding epic. Wave 3 is enhancement work.

### What ensures nothing gets lost

1. **This plan is the map** — every historical doc is listed with its target artifact linkage
2. **Wave 1 is additive only** — no files are moved or deleted, only frontmatter is added
3. **Retroactive epics are clearly marked** — `status: done` with historical dates, not pretending to be new work
4. **Each wave is committed separately** — git history preserves the migration trail
5. **The roadmap is updated after each wave** — reflecting the new artifact landscape
