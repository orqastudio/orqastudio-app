## Decision

OrqaStudio has three layers of configurability:

### Core Types (Hardcoded — the types always exist)

These artifact TYPES are part of the framework. Every project has them. They cannot be removed.

**Principles**: Pillars, Vision, Personas, Grounding
**Discovery**: Ideas, Research, Wireframes
**Learning**: Rules, Lessons, Decisions, Skills, Agents

The types are hardcoded. The INSTANCES are project-specific — which rules, which skills, which agents exist is up to the project. A software project has coding standards rules. A consulting project has engagement rules. Different content, same type system.

### Core State Machines (Configured, enforced by the app)

Each core type has a state machine defining how its instances progress. This is configured in a source-of-truth file (not in AI rules) and enforced directly by the application code.

The app reads the state machine config and enforces it mechanically:
- Valid statuses for each type
- Valid transitions between statuses
- Automatic transitions when conditions are met
- Structural requirements (e.g., project must have at least one pillar)

This is app-level enforcement, not AI enforcement. The app doesn't read RULE-NNN files to know what transitions are valid — it reads the state machine config directly. Rules exist as documentation for humans to understand what the system does.

### Delivery Pipeline (Fully project-configurable)

The work pipeline — what types of work artifacts exist and how they flow:
- **Artifact types** — what kinds of work exist (epics/tasks/milestones or hypotheses/experiments/phases)
- **Type hierarchy** — parent-child relationships
- **Status vocabulary** — already configurable (AD-049)
- **Transitions** — already configurable (AD-050)

A software project uses milestones/epics/tasks. A research project uses hypotheses/experiments/observations. The delivery pipeline adapts; the thinking framework stays the same.

### The Boundary

| Layer | Types | Instances | State Machine | Enforcement |
|-------|-------|-----------|---------------|-------------|
| **Principles** | Hardcoded | Project-specific | Configured, app-enforced | App |
| **Discovery** | Hardcoded | Project-specific | Configured, app-enforced | App |
| **Learning** | Hardcoded | Project-specific | Configured, app-enforced | App |
| **Delivery** | Project-configurable | Project-specific | Project-configurable | App |

The distinction: core types exist because they're part of the structured thinking methodology. Delivery types exist because the project needs them for tracking work. Both have configurable state machines. Both are enforced by the app, not by AI rules.

## What Gets Configured

```json
{
  "delivery": {
    "types": [
      {
        "key": "milestone",
        "label": "Milestone",
        "path": ".orqa/delivery/milestones",
        "gate_field": "gate"
      },
      {
        "key": "epic",
        "label": "Epic",
        "path": ".orqa/delivery/epics",
        "parent": { "type": "milestone", "field": "milestone" }
      },
      {
        "key": "task",
        "label": "Task",
        "path": ".orqa/delivery/tasks",
        "parent": { "type": "epic", "field": "epic" }
      }
    ],
    "integrity_rules": [
      {
        "name": "parent-child-consistency",
        "description": "Child can't be further along status progression than parent",
        "severity": "warning"
      },
      {
        "name": "completed-parent",
        "description": "Completed parent must have all children completed",
        "severity": "warning"
      }
    ]
  },
  "statuses": [ ... ],  // Already configurable
}
```

## What The Code Does

The code is an enforcement engine for the delivery pipeline:
1. Reads `project.json` for delivery type definitions and hierarchy
2. Scans configured paths for artifact files
3. Validates parent-child relationships using configured hierarchy
4. Evaluates status consistency using configured status order
5. Proposes transitions using configured auto_rules
6. Surfaces integrity violations to the user

The code does NOT know what a "milestone" or "task" is. It knows there's a type with a configured path, optional parent relationship, and a set of statuses it can progress through.

## What The Code Does For Core

The governance framework has fixed types, fixed schemas, and fixed relationships. The code knows:
- Rules have enforcement entries
- Lessons have recurrence tracking and promotion
- Skills have layer and portability constraints
- Agents have capabilities and grounding

This is not configurable because the learning loop IS the product.

## Rationale

Every project is different in what work looks like. But every project benefits from the same structured thinking process: capture → explore → plan → execute → reflect → improve. The delivery pipeline (how work is tracked) adapts to the project. The governance framework (how thinking is structured) stays constant.

## Consequences

- `project.json` gains a `delivery` section with type definitions and hierarchy
- Hardcoded references to "epic", "milestone", "task" in integrity checks are replaced with config-driven hierarchy lookups
- Roadmap/kanban components read type hierarchy from config
- New projects define their own delivery pipeline types
- Core governance types remain fixed per RULE-044
