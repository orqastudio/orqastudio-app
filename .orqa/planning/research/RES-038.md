## Current State

The ArtifactNav component has a text-only `SearchInput` (`h-7`, `size="xs"`) in a `border-b border-border p-2` container at the top. Below it, a `ScrollArea` renders either a tree view (collapsible folders with leaf items) or a flat list of `ArtifactListItem` components.

The NavSubPanel header is `h-10` with `text-xs font-semibold uppercase tracking-wide text-muted-foreground`. The Breadcrumb bar in the ArtifactViewer is also `h-10`. These establish the visual rhythm — `h-10` (40px) is the standard fixed-height bar in the sidebar.

The current `SearchInput` is `h-7` (28px) inside `p-2` padding, making the total bar ~44px — close to `h-10` but inconsistent.

## Design Decision: Toolbar Replaces Search

The text filter input is removed from the artifact panel. In its place: a compact icon-based toolbar at exactly `h-10` to match the NavSubPanel header and breadcrumb bar heights.

Search becomes a global AI-driven feature in the ActivityBar — it leaves the artifact panel entirely.

### Icon Buttons

Each button is a ghost icon button:

- **Size**: `h-7 w-7` (28px) — compact within the `h-10` bar
- **Style**: `rounded-md text-muted-foreground hover:bg-accent hover:text-foreground`
- **Icon size**: `h-3.5 w-3.5` — matches the `SearchIcon` size from the old bar
- **Active state**: When non-default sort or active filters exist, the button shows a small indicator:
  - **Sort**: `ArrowUpDownIcon` — becomes `text-foreground` when non-default sort is active
  - **Filter**: `FilterIcon` — shows a tiny `h-1.5 w-1.5 rounded-full bg-primary` dot in the top-right corner when filters are active (like a notification badge)

## Filter Dropdown Design

The filter dropdown is the most UX-critical element. It must feel structured and scannable, not like a generic checkbox list.

**Component choice: Popover, not DropdownMenu.** The existing `SessionDropdown.svelte` establishes a pattern of using `Popover` for rich interactive panels (with search inputs, scroll areas, custom list rendering). `DropdownMenu` is better for simple action menus. The filter panel is a rich panel — it uses `Popover`.

### Visual Hierarchy

```
┌─────────────────────────────────────┐
```

### Design Details

**Container:**
- `Popover.Content` with `w-56` width (224px)
- `align="start"` — aligns to the left edge of the filter button
- `sideOffset={4}` — small gap from toolbar
- Scrollable if content exceeds viewport: `max-h-[calc(100vh-200px)] overflow-y-auto`
- Uses the same Popover pattern as `SessionDropdown.svelte` — proven in the codebase for rich interactive panels

**Section Headings:**
- Custom `<p>` element: `text-[10px] font-semibold uppercase tracking-widest text-muted-foreground/60 px-2 pt-3 pb-1`
- Designed to be smaller and more recessive than list items — they are category labels, not items themselves

**Checkbox Items:**
- Custom checkbox rows (not DropdownMenu.CheckboxItem — we're in a Popover) with `h-8 px-2 text-xs`
- Use shadcn `Checkbox` component inline
- Left: checkbox indicator (built into shadcn CheckboxItem)
- Middle: status dot (`h-2 w-2 rounded-full`) using existing `StatusIndicator` colour system + label text
- Right: count badge in `text-muted-foreground ml-auto tabular-nums`
- The count updates live as other filters change (cross-filtering)

**Status Dot Colours** (reuse existing `StatusIndicator` `GROUP_DOT_CLASSES`):
- Draft/Todo/Captured/Planning/Proposed → `bg-zinc-400` (gray)
- In-progress/Exploring/Active/Ready → `bg-blue-500`
- Done/Complete/Accepted → `bg-emerald-500`
- Review/Recurring → `bg-amber-500`
- Promoted → `bg-purple-500`
- Superseded/Deprecated/Archived/Surpassed → `bg-red-400`

**Priority Colours:**
- P1 → `text-red-500 font-medium`
- P2 → `text-amber-500`
- P3 → `text-muted-foreground`

**Separators:**
- `DropdownMenu.Separator` between sections
- Thin: `h-px bg-border my-1`

**Clear Action:**
- Only visible when any filter is active
- `text-xs text-primary hover:underline cursor-pointer px-2 py-1.5`
- Positioned at the bottom, after the last section separator

### Dynamic Sections

The filter dropdown only shows sections relevant to the current artifact type. Section visibility is determined by inspecting the DocNode frontmatter fields available:

| Section | Shows When | Values From |
|---------|-----------|-------------|
| Status | Always (all types have `status`) | Status enum from schema.json for this type |
| Priority | Type has `priority` field (Epics only currently) | `[P1, P2, P3]` |
| Layer | Type has `layer` field (Rules, Skills, Agents, Decisions) | `[core, project, plugin, community, user]` |

Future: additional sections can be added without code changes — the system reads available fields from the schema and generates sections dynamically. But for Phase 1, the three above cover the useful cases.

### Count Calculation

Counts are calculated from the raw (unfiltered) node list for each filter value. This shows the total available items per status, not the currently-visible count. This matches common filter UX (like e-commerce faceted search).

When filters from OTHER sections are active, counts within each section reflect the cross-filtered result. Example: if "P1" priority filter is active, the status section counts show only P1 items per status.

## Sort Dropdown Design

The sort dropdown is simpler — a single-select list. This uses `DropdownMenu` (not Popover) since it's a standard radio-select menu without complex layout needs.

```
┌─────────────────────────────────────┐
│  SORT BY                            │  ← Section heading (same style as filter)
│  ┌─────────────────────────────┐    │
│  │ ✓ Title (A → Z)            │    │  ← Radio items with check on active
│  │   Title (Z → A)            │    │
│  │   Created (newest)         │    │
│  │   Created (oldest)         │    │
│  │   Updated (newest)         │    │
│  │   Updated (oldest)         │    │
│  └─────────────────────────────┘    │
│  ─────────────────────────────────  │
│  GROUP BY                           │
│  ┌─────────────────────────────┐    │
│  │ ✓ None                     │    │
│  │   Status                   │    │
│  └─────────────────────────────┘    │
│  ─────────────────────────────────  │
│  [Reset to default]                 │  ← Only when non-default
└─────────────────────────────────────┘
```

### Design Details

**Container:**
- `DropdownMenu.Content` with `w-48` (192px) — narrower than filter dropdown
- Same `align="start"`, `sideOffset={4}`

**Sort Section:**
- `DropdownMenu.RadioGroup` with `DropdownMenu.RadioItem` for each option
- Active sort shown with `CheckIcon` on the left (built into RadioItem)
- Items: `h-8 px-2 text-xs`
- Options are always the same set — they work generically across all types:
  - Title (A → Z) / Title (Z → A)
  - Created (newest first) / Created (oldest first)
  - Updated (newest first) / Updated (oldest first)
  - Status / Priority (only shown when the type has these fields)

**Group Section:**
- `DropdownMenu.RadioGroup` for group-by (only one active grouping)
- Options: None, Status (always available), Priority (when type has it)
- When a group-by is active, the artifact list renders with section headers instead of a flat list

**Reset Action:**
- Same pattern as "Clear all" in filter dropdown
- Only visible when current sort/group differs from `_navigation.json` defaults

## `_navigation.json` Data Model

### File Location

`_navigation.json` lives in the artifact TYPE directory (not the group directory). Each type can have its own navigation config:

```
.orqa/planning/epics/_navigation.json       ← epic-specific defaults
.orqa/planning/tasks/_navigation.json       ← task-specific defaults
.orqa/governance/rules/_navigation.json     ← rule-specific defaults
.orqa/documentation/_navigation.json        ← custom book layout for docs
```

### Schema

```jsonc
{
  // Standard sort/filter/group defaults — mutually exclusive with "layout"
  "defaults": {
    "sort": {
      "field": "updated",         // "title" | "created" | "updated" | "status" | "priority"
      "direction": "desc"         // "asc" | "desc"
    },
    "group": null,                // null | "status" | "priority"
    "filters": {                  // pre-set filters (empty = no filters)
      "status": ["draft", "ready", "in-progress", "review"]
    }
  },

  // Custom layout — mutually exclusive with "defaults"
  // When set, the artifact list renders sections from this config
  // instead of using sort/filter on the raw node list
  "layout": null
}
```

**Custom layout schema (for Documentation):**

```jsonc
{
  "defaults": null,
  "layout": {
    "sections": [
      {
        "label": "Product",
        "description": "Vision, governance, and product direction",
        "items": ["product/*"]        // glob patterns matching DocNode paths
      },
      {
        "label": "Architecture",
        "items": ["architecture/*"]
      },
      {
        "label": "Development",
        "items": ["development/*"]
      },
      {
        "label": "Process",
        "items": ["process/*"]
      },
      {
        "label": "User Interface",
        "items": ["ui/*", "wireframes/*"]
      },
      {
        "label": "User Guide",
        "items": ["guide/*"]
      }
    ],
    "uncategorized": "append"         // "append" | "hide" — what to do with items not matching any section
  }
}
```

### Scanner Integration

The Rust artifact scanner already reads `README.md` from each directory. Extending it to also read `_navigation.json` is straightforward:

1. In `artifact_reader.rs`, after reading README frontmatter for a NavType, check for `_navigation.json` in the same directory
2. If found, parse it and include in the `NavType` struct as an optional `navigation_config` field
3. The frontend receives this config alongside the nodes and applies defaults on first render

### Rust Type

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct NavigationConfig {
    pub defaults: Option<NavigationDefaults>,
    pub layout: Option<NavigationLayout>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct NavigationDefaults {
    pub sort: Option<SortConfig>,
    pub group: Option<String>,
    pub filters: Option<HashMap<String, Vec<String>>>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct SortConfig {
    pub field: String,
    pub direction: String,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct NavigationLayout {
    pub sections: Vec<LayoutSection>,
    pub uncategorized: Option<String>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct LayoutSection {
    pub label: String,
    pub description: Option<String>,
    pub items: Vec<String>,
}
```

### TypeScript Type

```typescript
export interface NavigationConfig {
  defaults: NavigationDefaults | null;
  layout: NavigationLayout | null;
}

export interface NavigationDefaults {
  sort: { field: string; direction: "asc" | "desc" } | null;

}

    label: string;
```

## Frontend State Model

### ArtifactViewState

```typescript
interface ArtifactViewState {
  sort: { field: string; direction: "asc" | "desc" };
  group: string | null;
  filters: Record<string, string[]>;  // field -> selected values
}
```

### Storage

- **Defaults**: Loaded from `_navigation.json` via the NavType's `navigation_config` field
- **User overrides**: Stored in the navigation store, keyed by artifact type key (e.g. `"epics"`, `"rules"`)
- **Persistence**: User overrides are session-only (reset on app restart). The `_navigation.json` file is the persistent config.
- **Future**: Could persist user overrides to SQLite if users want their preferences to survive restarts

### State Hierarchy

```
_navigation.json defaults → navigation store per-type state → user interactive overrides
```

The store initialises from `_navigation.json` defaults. User interactions update the store. "Reset to default" restores the `_navigation.json` state.

## Sorting & Filtering Logic (Client-Side)

All sorting and filtering happens client-side on the DocNode array. No new backend commands needed.

### Filtering

```typescript
function applyFilters(nodes: DocNode[], filters: Record<string, string[]>): DocNode[] {
  return nodes.filter(node => {
    for (const [field, values] of Object.entries(filters)) {
      if (values.length === 0) continue;
      const nodeValue = getNodeFieldValue(node, field);
      if (nodeValue && !values.includes(nodeValue)) return false;
    }
    return true;
  });
}
```

`getNodeFieldValue` extracts a field value from the DocNode. Currently DocNode only exposes `status`. To support filtering by `priority`, `layer`, etc., DocNode needs a `frontmatter` field — a `Record<string, unknown>` carrying the parsed YAML fields.

**Key change**: The Rust scanner must include frontmatter fields in DocNode for leaf nodes. This is a small backend change — the scanner already parses frontmatter for `status` and `description`; it just needs to include additional fields.

### Sorting

```typescript
function applySort(nodes: DocNode[], sort: SortConfig): DocNode[] {
  return [...nodes].sort((a, b) => {
    const aVal = getNodeFieldValue(a, sort.field);
    const bVal = getNodeFieldValue(b, sort.field);
    // String comparison for title/status; date comparison for created/updated
    const cmp = compareValues(aVal, bVal, sort.field);
    return sort.direction === "asc" ? cmp : -cmp;
  });
}
```

### Grouping

When `group` is set (e.g. "status"), the sorted list is partitioned into sections:

```typescript
function applyGrouping(nodes: DocNode[], groupField: string): Map<string, DocNode[]> {
  const groups = new Map<string, DocNode[]>();
  for (const node of nodes) {
    const key = getNodeFieldValue(node, groupField) ?? "Other";
    if (!groups.has(key)) groups.set(key, []);
    groups.get(key)!.push(node);
  }
  return groups;
}
```

Grouped output renders with section headers (same style as tree folder headers: `text-xs font-semibold uppercase tracking-wide text-muted-foreground`).

## DocNode Frontmatter Extension

The DocNode type needs a frontmatter bag:

```typescript
export interface DocNode {
  label: string;
  path: string | null;
  children: DocNode[] | null;
  status?: string | null;
  description?: string | null;
  icon?: string | null;
  /** Additional frontmatter fields for sorting/filtering */
  frontmatter?: Record<string, unknown>;
}
```

The Rust backend populates this with fields that are useful for sorting/filtering: `priority`, `layer`, `created`, `updated`, `milestone`, `epic`. The exact fields depend on the artifact type — but since the system is generic, the frontend reads whatever is provided.

## Implementation Phases

### Phase 1a: Backend — NavType navigation_config + DocNode frontmatter

1. Add `NavigationConfig` struct to Rust domain types
2. Read `_navigation.json` in the scanner alongside README.md
3. Include `navigation_config: Option<NavigationConfig>` in NavType
4. Extend DocNode to carry `frontmatter: Option<HashMap<String, serde_json::Value>>`
5. Populate frontmatter fields during scanning (priority, layer, created, updated, etc.)

### Phase 1b: Frontend — ArtifactToolbar + Sort/Filter Dropdowns

1. Create `ArtifactToolbar.svelte` — `h-10` bar with sort + filter icon buttons + count
2. Create `FilterPanel.svelte` — Popover-based sectioned checkbox panel (same pattern as SessionDropdown)
3. Create `SortDropdown.svelte` — DropdownMenu-based radio-style sort + group dropdown
4. Add `ArtifactViewState` to navigation store, keyed per artifact type
5. Wire filtering/sorting logic into ArtifactNav (replace SearchInput, apply transforms)
6. Support grouped rendering with section headers

### Phase 1c: `_navigation.json` files + Custom Layout

1. Create `_navigation.json` for key artifact types with sensible defaults
2. Implement custom layout rendering for Documentation
3. Toolbar adapts when layout mode is active (hides sort/filter, shows layout indicator)

## Open Questions

1. **Tree vs flat with filtering**: The current ArtifactNav has two modes — tree (for docs with subdirectories) and flat list. Sorting/filtering on a tree is complex. Should tree-mode types always use custom layout instead of sort/filter? Probably yes — Documentation already needs book-order layout, and it's the only tree-mode type currently.

2. **Frontmatter in DocNode**: How many fields to include? Minimal (status, priority, layer, created, updated) or everything? Recommend minimal — only fields that the filter/sort system uses.

3. **Cross-filter counts**: Should filter counts reflect cross-filtering (counts update as other sections filter) or always show totals? Cross-filtering is more useful but more complex. Recommend: totals for Phase 1, cross-filtering as enhancement.
